## 概述

VillageSQL 遵循清晰的版本控制和兼容性策略，帮助您在部署和升级时做出明智决策。

## VillageSQL 版本控制

VillageSQL 使用 [语义化版本控制](https://semver.org/) (SemVer) 来传达每次发布中变更的性质和影响。

### 版本格式

版本遵循格式：**MAJOR.MINOR.PATCH**

```
0.0.1
│ │ │
│ │ └─ PATCH version
│ └─── MINOR version
└───── MAJOR version
```

### 版本递增规则

- **MAJOR 版本**（例如 1.0.0 → 2.0.0）：可能需要修改代码或数据库迁移的破坏性变更
- **MINOR 版本**（例如 0.1.0 → 0.2.0）：以向后兼容方式添加的新功能和特性
- **PATCH 版本**（例如 0.0.1 → 0.0.2）：向后兼容的错误修复和小幅改进

<Note>
  VillageSQL 目前处于 1.0 之前的开发阶段（版本 0.x.x）。在此阶段，API 和扩展接口可能会更频繁地变化，以稳定平台。
</Note>

### 开发版本

带有预发布后缀的版本（例如 `0.0.4-dev`）是开发版本。默认情况下，服务器会阻止从使用开发版本初始化的数据库进行升级。开发版本未经升级兼容性测试，可能包含破坏性架构或协议变更。

要允许此操作，请使用 `--villagesql-allow-unsafe-dev-upgrade` 启动服务器：

```bash
mysqld --villagesql-allow-unsafe-dev-upgrade --datadir=/path/to/data
```

未指定该标志时，如果检测到开发版本数据库，服务器将报错退出：

```
Upgrading from a development version (0.0.4-dev) is not allowed. Use --villagesql-allow-unsafe-dev-upgrade to permit this.
```

指定该标志后，服务器会记录警告并继续运行：

```
Upgrading from a development version (0.0.4-dev); --villagesql-allow-unsafe-dev-upgrade was specified.
```

如果指定该标志但当前版本不是开发版本，服务器将报错退出：

```
--villagesql-allow-unsafe-dev-upgrade specified, but current version (0.0.3) is not a development version.
```

或如果未执行架构升级：

```
--villagesql-allow-unsafe-dev-upgrade specified, but no upgrade is being performed.
```

## 扩展兼容性

VillageSQL 扩展是版本特定的。为某个 VillageSQL 版本构建的扩展可能无法与另一个版本兼容。

升级 VillageSQL 时：

1. 检查新版本的扩展兼容性
2. 根据需要更新或重新构建扩展
3. 在生产部署前于开发环境中测试扩展

有关详细升级流程，请参阅 [升级指南](/guides/upgrade)。

## 0.0.3 版本 SDK 变更

**协议 2 扩展必须重新构建。** VEF 协议 3 自 v0.0.4 起稳定，服务器现在在加载时拒绝协议 2 扩展。请使用类型化包装器（`StringArg`、`IntArg`、`StringResult` 等）基于协议 3 API 重新构建。

**`vef_version_t::extra` 类型修正为 `const char*`。** 该字段在 0.0.3 中为 `unsigned char*`。大多数扩展不直接访问它——它由 SDK 从 `manifest.json` 的版本字段填充。如果您的扩展直接访问，请在使用 0.0.4 SDK 编译时移除任何 `(unsigned char *)` 类型转换。
