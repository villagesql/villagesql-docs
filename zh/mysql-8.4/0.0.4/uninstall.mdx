title: "卸载扩展"
description: "安全移除 VillageSQL 扩展"
---

## 命令语法

```sql
UNINSTALL EXTENSION extension_name [VERSION 'version'];
```

当提供 `VERSION` 时，服务器会将其与已安装扩展的版本进行比较，若版本不匹配则中止操作。省略该子句以卸载当前安装的版本。

<Warning>
在卸载前，必须移除所有使用扩展中自定义类型的列、存储过程参数和 DECLARE 变量。卸载过程将随后移除扩展提供的所有函数和类型。
</Warning>

## 先决条件

- VillageSQL 的管理员访问权限
- 无使用扩展自定义类型的表
- 无使用扩展自定义类型的存储过程参数或 DECLARE 变量
- 扩展存储自定义数据时的数据备份

## 卸载步骤

### 1. 尝试卸载

运行卸载命令。若扩展无依赖项，将立即成功。若列依赖于它，错误信息将明确标识具体列：

```sql
UNINSTALL EXTENSION vsql_complex;
```

**若扩展存在列依赖**，将看到类似错误：
```
ERROR HY000: Cannot drop extension `vsql_complex` as 1 column(s) depend on it,
e.g. mydb.signals.impedance has type COMPLEX
```

删除或修改标识的列后重试：

```sql
-- Option 1: Drop the table
DROP TABLE mydb.signals;

-- Option 2: Alter to remove only the custom type column
ALTER TABLE mydb.signals DROP COLUMN impedance;
```

**若存储过程使用扩展的自定义类型**，将看到类似错误：
```
ERROR 3219 (HY000): Cannot uninstall extension 'vsql_complex': stored procedure mydb.transform_signal uses custom type COMPLEX
```

删除或修改存储过程以移除类型参数或变量后重试：

```sql
-- Option 1: Drop the stored procedure
DROP PROCEDURE mydb.transform_signal;

-- Option 2: Recreate the procedure without the custom type parameter
-- (replace the custom type with a compatible built-in type or remove the parameter)
```

### 2. 卸载扩展

```sql
UNINSTALL EXTENSION vsql_complex;
```

为防止意外卸载非预期版本（在 CI 或滚动升级后很有用），请包含 `VERSION` 子句：

```sql
UNINSTALL EXTENSION vsql_complex VERSION '1.2.0';
```

若 `1.2.0` 非已安装版本，服务器将返回错误并保留扩展：

```
ERROR 3219 (HY000): Cannot uninstall extension 'vsql_complex': installed
version is '0.0.1' but VERSION '1.2.0' was specified
```

**此操作将移除：**
- ✅ 扩展提供的所有 VDF（函数）
- ✅ 扩展注册的自定义类型定义
- ✅ 扩展注册记录
- ✅ 扩展的自定义列元数据
- ✅ `mysqld-auto.cnf` 中扩展的持久化系统变量（`SET PERSIST`）

**此操作不会移除：**
- ❌ `.veb` 文件仍保留在 `veb_dir` 中
- ❌ `{datadir}/.veb_expansion_cache/` 中的展开目录将临时保留
- ❌ 表数据（需先删除使用自定义类型的表）

<Note>
`{datadir}/.veb_expansion_cache/{extension}/{sha256}/` 中的展开目录将在服务器重启时自动清理。
</Note>

<Note>
`UNINSTALL EXTENSION` 同时会从 `mysqld-auto.cnf` 中移除扩展的所有持久化系统变量。若之前执行过 `SET PERSIST extensionname.varname = value`，卸载时将自动删除该条目。持久化值在服务器关闭期间**不会**被移除——它们会保留重启周期，使扩展在下次加载时可读取。
</Note>

### 3. 验证卸载

```sql
-- Check extension is gone
SELECT * FROM INFORMATION_SCHEMA.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_complex';
-- Should return no rows

-- Verify functions removed
SELECT complex_abs('(1.0,2.0)');
-- Error: FUNCTION test_db.complex_abs does not exist
```

## 常见错误

| 错误 | 原因 | 解决方案 |
|-------|-------|----------|
| `Cannot drop extension \`name\` as N column(s) depend on it, e.g. db.table.col has type TYPE` | 表包含使用扩展自定义类型的列 | 删除或修改标识的列后重试 |
| `Cannot uninstall extension 'name': stored procedure db.sp uses custom type TYPE` | 存储过程包含使用扩展自定义类型的参数或 DECLARE 变量 | 删除或修改存储过程以移除自定义类型使用后重试 |
| `Cannot uninstall extension 'name': installed version is 'X' but VERSION 'Y' was specified` | 指定了 `VERSION` 子句但版本不匹配 | 重新运行时不带 `VERSION`，或使用 `SELECT EXTENSION_VERSION FROM INFORMATION_SCHEMA.EXTENSIONS` 中显示的版本 |
| `Extension not found` | 扩展未安装 | 通过 `SELECT * FROM INFORMATION_SCHEMA.EXTENSIONS` 验证 |

## 可选：移除 .veb 文件

卸载后 `.veb` 文件仍保留。如需完全移除：

```bash
# Find veb_dir
mysql -u root -p -e "SHOW VARIABLES LIKE 'veb_dir';"

# Delete .veb file
sudo rm /path/to/veb_dir/vsql_complex.veb
```

<Note>
`.veb` 文件可安全保留用于未来重新安装。删除它不会影响服务器性能。
</Note>

## 重新安装

如需重新安装扩展，请再次使用 `INSTALL EXTENSION`（若 `.veb` 文件仍存在）：

```sql
INSTALL EXTENSION vsql_complex;
```

详情请参阅 [安装扩展](/mysql-8.4/0.0.4/install)。
