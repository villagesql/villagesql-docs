The VillageSQL Extension Framework (VEF) provides a C++17 API for writing custom SQL functions and types that run natively inside VillageSQL. Extensions are compiled to `.veb` bundles and installed with `INSTALL EXTENSION` — no server source modification required.

## Installation

Extensions are installed after building them from source:

```sql
-- Install a compiled .veb bundle
INSTALL EXTENSION my_extension;

-- Verify installation
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'my_extension';
```

## Extension Structure

A VEF extension has three required parts:

| File | Purpose |
| :--- | :--- |
| `manifest.json` | Extension name, version, and dependencies |
| `CMakeLists.txt` | Build configuration referencing `VillageSQL_BUILD_DIR` |
| `src/*.cc` | C++17 implementation using the VEF API |

## Implementation: Building a Custom Function

The VEF API follows a consistent pattern: write an implementation function, then register it with a name and type signature.

**C++ implementation (`src/my_func.cc`):**

```cpp
#include <vef.h>

void my_upper_impl(vef_context_t* ctx,
                   vef_invalue_t* arg,
                   vef_vdf_result_t* result) {
    if (arg->is_null) {
        result->type = IS_NULL;
        return;
    }
    std::string input(arg->str_value, arg->str_len);
    std::transform(input.begin(), input.end(), input.begin(), ::toupper);
    strncpy(result->str_buf, input.c_str(), result->max_str_len);
    result->str_len = input.size();
}

VEF_GENERATE_ENTRY_POINTS(
    make_extension("my_extension", "1.0.0")
        .func(make_func<&my_upper_impl>("my_upper")
            .returns(STRING)
            .param(STRING)
            .buffer_size(1024)
            .build())
)
```

**Build and install:**

```bash
mkdir build && cd build
cmake .. -DVillageSQL_BUILD_DIR=/path/to/villagesql/build
make
```

Copy the compiled `.veb` bundle to the server's VEB directory. Get the path from the running server first:

```sql
-- Check where the server loads .veb files from
SELECT @@veb_dir;
```

```bash
# Copy the bundle to that path (replace /var/lib/villagesql/extensions with your @@veb_dir value)
cp my_extension.veb /var/lib/villagesql/extensions/
```

```sql
INSTALL EXTENSION my_extension;
SELECT my_upper('hello world');
-- Returns: 'HELLO WORLD'
```

## Comparison: VillageSQL VEF vs. Standard MySQL 8.4

| Feature | Standard MySQL 8.4 LTS | VillageSQL (VEF SDK) |
| :--- | :--- | :--- |
| **Custom Types** | Not supported without patching server source. | Native custom type support via VEF type registration. |
| **Memory Management** | Manual `malloc`/`free` handling against MySQL internals. | VEF manages memory lifecycle automatically. |
| **Distribution** | Loose `.so` files without schema or dependency metadata. | Packaged as `.veb` bundles with manifest and schema included. |
| **Installation** | Plugin install requires server restart in some cases. | `INSTALL EXTENSION` is hot-loaded; no restart needed. |

## Frequently Asked Questions

#### Do I need to modify the VillageSQL server source to add a function?
No. VEF extensions compile to shared objects that are loaded at runtime via `INSTALL EXTENSION`. The server source is not touched.

#### Can I write extensions in languages other than C++?
The VEF SDK targets C++17. Other languages can interoperate via a C-compatible ABI, but the primary interface is C++.

#### What is the difference between VEF and MySQL's built-in UDF mechanism?
MySQL UDFs use a C-level API that is not type-safe and requires manual memory management against MySQL internals. The VEF SDK provides a typed C++17 API with memory management, dependency resolution, and packaging built in.

#### Where can I find a complete extension template?
See the `vsql-extension-template` repository in the VillageSQL GitHub organization. It includes a working "Hello World" extension, `CMakeLists.txt`, `manifest.json`, and MTR tests.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `Extension not found` on INSTALL | `.veb` file not in `@@veb_dir` | Verify `SELECT @@veb_dir` and copy the `.veb` file there |
| CMake can't find VillageSQL | `VillageSQL_BUILD_DIR` not set | Pass `-DVillageSQL_BUILD_DIR=/path/to/server/build` to cmake |
| Function returns unexpected `NULL` | `is_null` not checked | Check `if (arg->is_null) { result->type = IS_NULL; return; }` first |
| Build succeeds but function missing after install | Entry point name mismatch | Confirm the name in `make_func<...>("name")` matches what you're calling |
