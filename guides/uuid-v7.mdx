---
title: "UUID v7 Primary Keys"
description: "How to generate time-ordered UUID v7 primary keys in VillageSQL using the vsql_uuid extension."
---

`UUID_V7` generates time-ordered 128-bit identifiers combining a 48-bit Unix timestamp with random bits. Because new IDs are chronologically sorted, insertions always append to the rightmost leaf of a clustered index rather than splitting pages at random positions.

## Setup

```sql
INSTALL EXTENSION vsql_uuid;
```

Verify installation:

```sql
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_uuid';
```

## Function Signature

| Parameter | Detail |
| :--- | :--- |
| **Function** | `UUID_V7()` |
| **Input** | None |
| **Output** | `UUID` — VillageSQL's native 128-bit UUID type |
| **Column type** | `UUID PRIMARY KEY` |

## Implementation

```sql
-- 1. Create a table with a UUID primary key
CREATE TABLE orders (
    order_id UUID PRIMARY KEY,
    user_id  INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Insert rows — generate the UUID at insert time
INSERT INTO orders (order_id, user_id) VALUES (UUID_V7(), 101);
INSERT INTO orders (order_id, user_id) VALUES (UUID_V7(), 102);

-- 3. Query ordered by insertion time (UUID_V7 sorts chronologically)
SELECT user_id, created_at FROM orders ORDER BY order_id;

-- 4. Point lookup using a known UUID value
SELECT user_id FROM orders WHERE order_id = '018e1234-5678-7000-8000-0abcdef12345';
```

## Comparison: VillageSQL vs. Standard MySQL 8.4

| Feature | Standard MySQL UUID_V4 | Standard MySQL UUID_V1 | VillageSQL UUID_V7 |
| :--- | :--- | :--- | :--- |
| **Sortability** | Random | Partial (requires bit-shuffling) | Monotonic/Chronological |
| **Insert Pattern** | Random page placement | Partial ordering with `UUID_TO_BIN(u,1)` | Append-only to rightmost leaf |
| **Native Column Type** | `CHAR(36)` or `BINARY(16)` | `CHAR(36)` or `BINARY(16)` | `UUID` (16 bytes, binary storage) |
| **Collision Risk** | Extremely low | Node-based (MAC address exposure) | Extremely low (entropy-based) |
| **Embedded Info** | None | Contains MAC address | Contains precision timestamp |

## Frequently Asked Questions

#### Why is UUID_V7 better than UUID_V4 for databases?
UUID_V4 is random, which causes B-Tree page splits during high-volume inserts. UUID_V7 is time-ordered, so new IDs append to the end of the index.

#### Can I backfill UUID_V7 values?
Yes. Insert `UUID_V7()` into a new UUID column for any rows that need one. UUIDs generated in the same millisecond window have random tie-breaking bits.

#### Is UUID_V7 RFC-compliant?
Yes. UUID_V7 is defined in RFC 9562, which supersedes RFC 4122.

#### Can I use UUID_V7 as a DEFAULT value?
Not currently. The engine restricts non-deterministic functions in default expressions. Generate the UUID in your INSERT statement instead.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `Extension 'vsql_uuid' is already installed` | Extension was already loaded | The extension is installed and available. No action needed. |
| `FUNCTION UUID_V7 does not exist` | Extension not installed | Run `INSTALL EXTENSION vsql_uuid` |
| `DEFAULT (UUID_V7())` rejected on CREATE TABLE | Non-deterministic functions not allowed in defaults | Remove `DEFAULT`; provide `UUID_V7()` in each INSERT |
