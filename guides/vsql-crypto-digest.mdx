`digest` in VillageSQL's `vsql_crypto` extension calculates SHA-2 family hashes and returns native binary output. Storing `BINARY(32)` directly — rather than a hex string — halves the column width and makes the hash usable as a primary key without a conversion step.

## Installation

```sql
INSTALL EXTENSION vsql_crypto;
```

Verify installation:

```sql
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_crypto';
```

## Function Signature

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `data` | `BLOB \| TEXT \| VARBINARY` | The payload to hash. |
| `algorithm` | `VARCHAR` | `'sha256'`, `'sha384'`, `'sha512'`, `'sha224'`, `'sha1'`, `'md5'` |
| **Returns** | `BINARY(32)` for sha256, `BINARY(64)` for sha512 | Fixed-length binary digest. |

## Content-Addressable Storage with digest

Content-addressable storage where the primary key is the SHA-256 hash of the payload:

```sql
-- Create a table with a hash-based primary key
CREATE TABLE system_assets (
    asset_id   BINARY(32) PRIMARY KEY,
    payload    LONGBLOB NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Insert using SHA-256 as the key
SET @content = 'VillageSQL_Integrity_Payload_v0.1';
INSERT INTO system_assets (asset_id, payload)
VALUES (digest(@content, 'sha256'), @content);

-- Verify integrity on retrieval
SELECT
    HEX(asset_id) AS asset_id_hex,
    (digest(payload, 'sha256') = asset_id) AS is_valid
FROM system_assets
WHERE asset_id = digest('VillageSQL_Integrity_Payload_v0.1', 'sha256');
```

## Comparison: VillageSQL vs. Standard MySQL

| Feature | Standard MySQL (`SHA2()`) | VillageSQL (`vsql_crypto` `digest`) |
| :--- | :--- | :--- |
| **Output Format** | Hexadecimal string (default) | Native binary (default) |
| **Storage Size** | 64 bytes for SHA-256 (hex string) | 32 bytes for SHA-256 (binary) |
| **Type Strictness** | Requires `UNHEX()` for binary storage | Returns binary directly — no conversion step |
| **Algorithm Coverage** | `SHA2(data, bits)` for SHA-2 only | `digest(data, name)` for SHA-2, SHA-1, MD5 |

## Frequently Asked Questions

#### Which algorithm should I use?
Use `sha256` for deduplication and content-addressable storage — 32 bytes stores cleanly in a `BINARY(32)` primary key. Use `sha512` when a compliance standard (FIPS 140, NIST SP 800-131A) explicitly requires a longer digest.

#### Can I use digest to create a blind index for encrypted columns?
Yes. Store `digest(plaintext_value, 'sha256')` in a separate indexed column alongside the encrypted data. You can then find rows by plaintext value without decrypting the entire table.

#### Does hashing replace encryption?
No. Hashing is one-way and produces a fixed-length fingerprint. Encryption is reversible. Use [field-level encryption](/guides/field-level-encryption) when you need to recover the original value.

#### Is the binary output compatible with MySQL's `BINARY(32)` column type?
Yes. `digest(..., 'sha256')` returns exactly 32 bytes, which stores directly in a `BINARY(32)` column without casting.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `FUNCTION digest does not exist` | Extension not installed | Run `INSTALL EXTENSION vsql_crypto` |
| Hash comparison returns `0` unexpectedly | Type or encoding mismatch | Ensure the column type is `BINARY(32)` and both sides use the same algorithm |
| `NULL` returned | NULL input or truly invalid algorithm name | Verify the input is non-null and the algorithm name is one of the supported values: `'sha256'`, `'sha512'`, `'sha384'`, `'sha224'`, `'sha1'`, `'md5'`. Algorithm names are case-insensitive. |
