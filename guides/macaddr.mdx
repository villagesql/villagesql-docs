---
title: "Storing MAC Addresses"
description: "How to store, index, and query MAC addresses in VillageSQL using the native MACADDR data type."
---

VillageSQL's `MACADDR` data type stores IEEE 802 MAC addresses as fixed 6-byte binaries with native EUI-48 validation. The database enforces address format on insert, supports indexing, and preserves human-readable output without needing a separate display column.

## Setup

```sql
INSTALL EXTENSION vsql_network_address;
```

Verify installation:

```sql
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_network_address';
```

## Type Reference

| Property | Detail |
| :--- | :--- |
| **Type Name** | `MACADDR` |
| **Accepted Input Formats** | `XX:XX:XX:XX:XX:XX`, `XX-XX-XX-XX-XX-XX`, `XXXX.XXXX.XXXX` |
| **Canonical Output** | Lowercase hex with colons: `aa:bb:cc:dd:ee:ff` |
| **Internal Storage** | 6-byte fixed-width binary |

## Implementation

```sql
-- Create a table using the native MACADDR type
CREATE TABLE network_logs (
    log_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    capture_time TIMESTAMP(6) DEFAULT CURRENT_TIMESTAMP(6),
    source_mac MACADDR NOT NULL,
    dest_mac MACADDR NOT NULL,
    payload_size INT,
    INDEX idx_src_mac (source_mac),
    INDEX idx_dest_mac (dest_mac)
) ENGINE=InnoDB;

-- Insert data using different accepted formats
INSERT INTO network_logs (source_mac, dest_mac, payload_size)
VALUES
    ('00:1A:2B:3C:4D:5E', '00-1A-2B-3C-4D-5F', 512),
    ('aa:bb:cc:dd:ee:ff', '1122.3344.5566', 1024);

-- Query by manufacturer prefix (OUI = first 3 bytes)
-- MACADDR compares by binary byte value, so colon-formatted strings
-- with the same first 3 bytes sort together in the index.
SELECT * FROM network_logs
WHERE source_mac >= '00:1A:2B:00:00:00'
  AND source_mac <= '00:1A:2B:FF:FF:FF';

-- Truncate to OUI prefix (first 3 bytes) for manufacturer grouping
SELECT macaddr_trunc(source_mac) AS oui_prefix, COUNT(*) AS count
FROM network_logs
GROUP BY macaddr_trunc(source_mac);
```

## Comparison: VillageSQL vs. Standard MySQL 8.4

| Feature | Standard MySQL 8.4 | VillageSQL (MACADDR) |
| :--- | :--- | :--- |
| **Storage** | 17–23 bytes (`VARCHAR`) | 6 bytes (fixed binary) |
| **Validation** | Manual (regex triggers or application logic) | Native EUI-48 enforcement on insert |
| **Index Behavior** | String comparison on `VARCHAR`; binary comparison on `BINARY(6)` | Integer-based comparison on fixed-width binary |
| **Human Readability** | Lost if stored as `BINARY(6)` | Preserved via native type display |
| **Multi-format Input** | Requires application-side normalization | All standard formats accepted natively |

## Frequently Asked Questions

#### Which MAC address formats does VillageSQL accept?
VillageSQL accepts colon-separated (`AA:BB:CC:DD:EE:FF`), hyphen-separated (`AA-BB-CC-DD-EE-FF`), and Cisco dot-notation (`AABB.CCDD.EEFF`). All are normalized to lowercase colon-separated format on storage.

#### What error appears when an invalid MAC address is inserted?
VillageSQL raises `ER_TRUNCATED_WRONG_VALUE_FOR_FIELD` with the invalid value and column name. The insert is rejected.

#### Can I index a MACADDR column?
Yes. Use a standard `INDEX` on the column. Because `MACADDR` uses fixed-width 6-byte storage, the B-Tree index is compact and well-ordered.

#### How do I query by vendor prefix (OUI)?
Use a range query against the binary ordering of the address:
```sql
SELECT * FROM network_logs
WHERE source_mac >= '00:1A:2B:00:00:00'
  AND source_mac <= '00:1A:2B:FF:FF:FF';
```
Or use `macaddr_trunc(column)` to group by OUI prefix.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `FUNCTION MACADDR does not exist` | Extension not installed | Run `INSTALL EXTENSION vsql_network_address` |
| `ER_TRUNCATED_WRONG_VALUE_FOR_FIELD` on insert | Invalid MAC address format | Check the value; ensure it uses a supported separator format |
| Range query returns no rows | Address comparison order mismatch | Confirm both bounds use the same format and are valid MACADDR values |
