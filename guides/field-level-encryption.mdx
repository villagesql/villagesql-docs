`encrypt` and `decrypt` in VillageSQL's `vsql_crypto` extension let you encrypt individual column values at insert time and decrypt them on retrieval. Encryption keys are passed as SQL parameters — keep them in application configuration or session variables, not hardcoded in queries.

## Installation

```sql
INSTALL EXTENSION vsql_crypto;
```

Verify installation:

```sql
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_crypto';
```

## Function Signatures

| Function | Parameters | Returns |
| :--- | :--- | :--- |
| `encrypt(data, key, type)` | `data` TEXT/BLOB, `key` VARCHAR, `type` VARCHAR | `VARBINARY` — encrypted payload |
| `decrypt(data, key, type)` | `data` VARBINARY, `key` VARCHAR, `type` VARCHAR | `VARBINARY` — plaintext |

**Supported `type` values:** `'aes'`, `'aes-256'`

## Encrypting and Decrypting Column Values

```sql
-- 1. Create a table with binary columns for encrypted fields
CREATE TABLE users_pii (
    user_id        INT PRIMARY KEY,
    email_encrypted VARBINARY(255),
    ssn_encrypted   VARBINARY(255)
);

-- 2. Encrypt on insert (key managed by the application)
SET @enc_key = 'sixteen-byte-key';
INSERT INTO users_pii (user_id, email_encrypted, ssn_encrypted)
VALUES (
    101,
    encrypt('dev@example.com', @enc_key, 'aes'),
    encrypt('999-00-1234',     @enc_key, 'aes')
);

-- 3. Decrypt on retrieval
SET @enc_key = 'sixteen-byte-key';
SELECT
    user_id,
    CAST(decrypt(email_encrypted, @enc_key, 'aes') AS CHAR) AS email
FROM users_pii
WHERE user_id = 101;
```

## Comparison: VillageSQL vs. Standard MySQL 8.4

| Feature | MySQL `AES_ENCRYPT` | VillageSQL `vsql_crypto` `encrypt` |
| :--- | :--- | :--- |
| **Algorithm** | Configurable; defaults to ECB (no authentication). | AES with configurable mode. |
| **pgcrypto Compatibility** | None. | PostgreSQL pgcrypto-compatible API. |
| **Output** | `VARBINARY`. | `VARBINARY`. |
| **Key Handling** | Key passed as string literal. | Key passed as string parameter — store it in a session variable, not in the query itself. |

## Frequently Asked Questions

#### How should I manage the encryption key?
Store the key in your application configuration and pass it as a session variable (`SET @enc_key = ...`) at the start of each session. Never embed raw key literals in stored procedures or views where they might appear in query logs.

#### Can I search encrypted columns?
Encrypted columns can't be used for equality or range lookups directly. For searchable encrypted data, use [vsql_crypto digest](/guides/vsql-crypto-digest) to store a hash of the plaintext in a separate indexed column (a blind index).

#### What happens if I decrypt with the wrong key?
`decrypt` returns `NULL` — it does not raise an error. Always check for a NULL result and verify the output with application-layer validation when correctness matters.

#### What key length should I use?
For `'aes-256'`, use a 32-byte key. For `'aes'`, a 16-byte key. Shorter keys are padded; longer keys are truncated by the underlying OpenSSL implementation.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `FUNCTION encrypt does not exist` | Extension not installed | Run `INSTALL EXTENSION vsql_crypto` |
| Decrypted value is garbled | Wrong key used for decryption | Ensure the same key value is used for both encrypt and decrypt |
| `VARBINARY` column too small | Ciphertext is longer than plaintext | Size the column as `VARBINARY(255)` minimum; AES adds padding |
