VillageSQL's `COMPLEX` data type stores complex numbers $(a + bi)$ as a single typed column. Arithmetic — addition, multiplication, magnitude, conjugate — is handled by native SQL functions rather than manual two-column formulas.

## Installation

```sql
INSTALL EXTENSION vsql_complex;
```

Verify installation:

```sql
SELECT * FROM information_schema.EXTENSIONS WHERE EXTENSION_NAME = 'vsql_complex';
```

## Function Reference

| Function | Returns | Description |
| :--- | :--- | :--- |
| `complex_add(c1, c2)` | `COMPLEX` | Sum of two complex values |
| `complex_subtract(c1, c2)` | `COMPLEX` | Difference of two complex values |
| `complex_multiply(c1, c2)` | `COMPLEX` | Product of two complex values |
| `complex_divide(c1, c2)` | `COMPLEX` | Quotient of two complex values |
| `complex_abs(c)` | `DOUBLE` | Modulus (magnitude): `√(a² + b²)` |
| `complex_real(c)` | `DOUBLE` | Real component |
| `complex_imag(c)` | `DOUBLE` | Imaginary component |
| `complex_conjugate(c)` | `COMPLEX` | Conjugate: `a - bi` |

Complex values are inserted as string literals in `'(real,imag)'` format.

## Computing with COMPLEX Values

Example: computing signal magnitude in a digital signal processing (DSP) table.

```sql
-- Create a table with native COMPLEX support
CREATE TABLE signal_data (
    id INT PRIMARY KEY,
    frequency_bin INT,
    coefficient COMPLEX
);

-- Insert complex values using string literal format '(real,imag)'
INSERT INTO signal_data (id, frequency_bin, coefficient)
VALUES
    (1, 100, '(0.5,0.866)'),
    (2, 200, '(-0.707,0.707)');

-- Retrieve magnitude and components
SELECT
    id,
    coefficient,
    complex_real(coefficient)  AS real_part,
    complex_imag(coefficient)  AS imag_part,
    complex_abs(coefficient)   AS magnitude
FROM signal_data;

-- Scalar multiply (amplify signal by factor of 2)
SELECT
    id,
    complex_multiply(coefficient, '(2.0,0.0)') AS amplified,
    complex_abs(complex_multiply(coefficient, '(2.0,0.0)')) AS amplified_magnitude
FROM signal_data
WHERE complex_abs(coefficient) > 0.5;
```

## Comparison: VillageSQL vs. Standard MySQL 8.4

| Feature | Standard MySQL 8.4 | VillageSQL (`COMPLEX`) |
| :--- | :--- | :--- |
| **Data Storage** | Two separate `DOUBLE` columns for real and imaginary parts | Single native `COMPLEX` column |
| **Arithmetic** | Manual SQL expressions: `(a*c - b*d)` for multiplication | Native `complex_multiply`, `complex_add` functions |
| **Column Constraint** | No type-level constraint on the pair | Single column can be `NOT NULL`, indexed by derived value |
| **Readability** | Complex formulas in SQL | Function names express the operation directly |

## Frequently Asked Questions

#### How do I insert a complex value?
Use a string literal in `'(real,imag)'` format: `INSERT INTO t VALUES (1, '(3.0,4.0)')`. There is no constructor function — the string is parsed by the type system on insert.

#### Can I index a COMPLEX column?
You can create a functional index on a derived scalar value such as magnitude: `INDEX idx_magnitude ((complex_abs(coefficient)))`. Direct indexes on the `COMPLEX` column itself are not supported.

#### Can I compare two COMPLEX values with `=`?
Equality comparisons on floating-point values are unreliable regardless of storage type. Compare magnitudes or use a tolerance check: `ABS(complex_abs(c1) - complex_abs(c2)) < 0.0001`.

#### What precision does COMPLEX use internally?
Both the real and imaginary components are stored as IEEE 754 double-precision (64-bit) floats, matching MySQL's `DOUBLE` type.

## Troubleshooting

| Problem | Cause | Solution |
| :--- | :--- | :--- |
| `Extension 'vsql_complex' is already installed` | Extension was already loaded | The extension is installed and available. No action needed. |
| Insert fails with wrong value error | String not in `'(real,imag)'` format | Use parentheses and a comma: `'(1.0,2.0)'` |
| `complex_multiply` returns unexpected results | Complex arithmetic (not scalar) | Confirm both arguments use `'(real,imag)'` format |
| Magnitude filter returns no rows | `complex_abs` threshold too restrictive | Test with `SELECT complex_abs(coefficient) FROM signal_data` to inspect actual magnitudes |
